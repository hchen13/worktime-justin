- 类型: visual
- 优先级: P0
- 关联需求: WTJ-20260703-002 验收标准 5（移动宽度排版清晰）
- 前置: `python3 tests/visual/scripts/capture_docs.py`；判读前确认
  `tests/reports/docs-qc/GATE_FAILED` **不存在**（存在=硬门禁失败——含 segment
  触顶截断，截图已被移入 `quarantine/`，禁止判读）。advisory 探针
  （对比度/alt/console，归属 009/012）失败不影响本用例。
- 输入截图: `tests/reports/docs-qc/mobile-first-screen.png` + mobile 全套

  唯一权威**，严格按清单顺序逐张取图，不要 glob segments 目录。

## 测什么
390px 宽（手机竖屏）下整页是否仍是一个可用的阅读体验：卡片正确堆叠为单列、
表格在窄屏下可读（不逐字换行、不溢出）、导航可用、图片按宽度缩放不变形、
字号不至于过小。

## 怎么测
把 mobile 首屏 + 全部 segment 按顺序交给视觉子代理，执行下面 prompt。
（脚本层的水平溢出检查由 DOCQC-007 覆盖，这里专注"数值上没溢出但读起来难受"
的问题。）

## 视觉理解 Prompt

```
你在质检一个需求文档静态站的手机宽度（390px）排版。下面是首屏 + 按顺序切开的
整页截图。逐张查看后回答具体问题：

1. 多栏卡片是否已全部堆叠为单列？有没有两个卡片仍并排导致每个都过窄的情况？
   指出 segment 编号。
2. 页面里的表格在这个宽度下处于什么状态？三选一并给证据：
   a) 正常可读（每格文字成词成句）；
   b) 勉强可读但明显拥挤（出现整列逐字竖排换行、单元格只剩 1-2 个字宽）；
   c) 溢出或截断（有列看不到或文字被切）。
   把最窄一列的表头和一格内容抄出来。
3. 顶部导航在窄屏下是什么形态？是否能看出它可横向滚动或换行显示，而不是被硬裁切
   且没有任何可滚动迹象？
4. 产品 mock 截图是否按容器宽度等比缩放？有没有被拉伸变形、只显示一半、或超出
   卡片边缘？
5. 正文和列表字号在手机上是否还能舒适阅读？抄一段你认为最小的可读正文。
6. 有没有任何元素（长英文串、tag、代码路径如 .codex/skills/）撑破容器或触发
   明显的左右滚动迹象？

输出严格 JSON：
{
  "pass": true/false,
  "reason": "一句话",
  "evidence": ["每个问题的结论 + segment 编号 + 抄录文字"],
  "risks": ["b 级拥挤等不到 fail 的问题"]
}
判定规则：问题 2 为 c、问题 1/4/6 发现并排挤压、变形或撑破容器 → pass=false。
问题 2 为 b（如整列逐字竖排）记为 pass=false 当且仅当受影响的是承载需求信息的
主内容表；否则进 risks。问题 3 只要存在可用的导航形态即可。
```

## 怎么算过
视觉子代理 pass=true。已知基线（2026-07-03 seed draft）：鼠标反馈表在 390px 下
第一列"输入/移动/点击/拖拽"逐字竖排、四列均 1-4 字宽（b 级拥挤，承载需求信息），
按上述规则应判 pass=false —— 该问题已作为改进项反馈 TL 卡：移动端表格需要
overflow-x 容器或改为堆叠卡片。TL 交付正式版后本用例期望变为 pass=true。
